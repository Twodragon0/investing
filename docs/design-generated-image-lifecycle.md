# 설계 검토: 생성 이미지 수명 vs 포스트 영구 참조

> 작성 2026-06-10. 트리거: `cleanup-old-images.yml`(30일 정리)과 포스트의 영구 이미지 참조 충돌로
> 빌드 기준 1492개 중 1052개(70%) 포스트가 깨진 hero/og 이미지를 노출하던 문제.
> 1차 대응(레이아웃 존재 가드)은 커밋 `5c086d0cc`, `c3482a37d`로 완료 — 더 이상 깨진 이미지는 없다.
> 본 문서는 **그 이상의 근본 재설계** 옵션을 평가한다.

## 현황 (측정)

- 생성 이미지: **3844 파일 / 214MB** (≈30일 롤링 윈도우). 정리 워크플로우는 정상 작동 중.
- 증가율: **~120 파일/일** (≈40 논리 이미지 × {avif, webp, png} + thumb 변형).
- 정리 미적용 시: 워킹트리 **~2.6GB/년** 증가 + git 히스토리 영구 누적(blob은 정리해도 히스토리에 남음).
- 충돌의 본질: 포스트는 이미지를 **영구** 참조(front matter `image:`, hero, related-posts, cards,
  journal landing, og:image)하는데, 정리는 저장소 비대화를 막으려 **30일 후 삭제**한다.

## 1차 대응 상태 (완료, baseline)

레이아웃의 모든 생성 이미지 렌더 표면에 `site.static_files` 존재 가드 적용:
- `_includes/generated-picture.html` — 보편 가드(파일 없으면 미렌더), 7개 호출처 보호
- `_layouts/default.html` — og:image(page.image + 콘텐츠 폴백) / avif preload
- `_layouts/post.html` — hero / related-post / JSON-LD
- `_includes/post-card.html` — 카드 썸네일 → SVG 카테고리 아이콘 폴백
- 회귀 테스트 `tests/test_generated_image_guard.py`

**효과**: 깨진 이미지 139→0, og:image 404 1052→0. 저장소 규모 불변. **운영상 충분.**
**잔여 비용**: 30일 지난 포스트는 고유 hero/og 이미지를 잃고 기본 og + 카테고리 아이콘으로 표시(시각/소셜 다운그레이드, 랭킹 영향은 미미).

## 옵션 비교

| 옵션 | 설명 | 저장소 | 노력 | 옛 포스트 시각 | 비고 |
|---|---|---|---|---|---|
| **A. 현행 + 가드** (완료) | 30일 정리 유지, 누락은 우아하게 폴백 | 214MB 고정 ✓ | 0 (완료) | 기본 og/아이콘 | **현재 상태. 권장 baseline** |
| B. 참조 이미지 보존 | 정리에서 "포스트가 참조 중인 이미지" 제외 | ~2.6GB/년 ↑ ✗ | 소 | 완전 보존 | git 히스토리·클론 비대화. **비권장** |
| C. 썸네일만 장기 보존 | 30일 후 대형 hero/og는 삭제, 소형 `thumb-og-*`만 유지 | 완만 증가 | 중 | 카드/목록은 풍부, hero 없음 | 절충안. 정리 로직 분기 필요 |
| D. 외부 오브젝트 스토리지/CDN | 이미지를 git 밖(R2/S3/GH release/LFS)으로, 포스트는 절대 URL 참조 | git 최소 ✓✓ | 대 | 영구 보존 ✓ | **장기 정답**. 인프라(버킷·자격증명·업로드 스텝·URL 재작성) 필요 |
| E. 빌드 시 재생성 | 빌드에서 누락 이미지 재생성 | - | - | - | **불가**: 과거 시세/뉴스 원본이 시간의존·소멸. 재현 불가 |

## 권장안

1. **단기 (지금)**: **옵션 A 유지.** 깨진 이미지 0, 저장소 고정. 추가 작업 불필요.
2. **옛 포스트(에버그린/검색유입) 시각 보존이 가치 있다고 판단되면 → 옵션 D 채택.**
   - 저장소에서 `assets/images/generated/` 제거, 생성 시점에 R2/S3 등으로 업로드, 포스트는 CDN 절대 URL 참조.
   - 이미지 수명을 저장소 크기와 **분리** → 정리 워크플로우 자체가 불필요해짐.
   - 선결: 버킷·자격증명(시크릿), `image_generator` 업로드 경로, 포스트 생성기 URL 주입, 기존 포스트 URL 마이그레이션.
3. **D의 인프라가 부담이면 → 옵션 C를 중간 단계로.** `thumb-og-*`(파일당 ~20KB)만 30일 이후 유지 →
   목록/카드 시각 유지하며 증가율을 ~1/5로 억제. hero는 여전히 폴백.

## 결정 필요 항목 (사용자)

- 옛 포스트의 **고유 hero/og 이미지 보존**이 사업적으로 필요한가? (에버그린 콘텐츠 비중·소셜 재공유 빈도)
  - 아니오 → 옵션 A로 종결.
  - 예 → 옵션 D 채택, 별도 구현 캠페인(인프라+마이그레이션)으로 진행.
